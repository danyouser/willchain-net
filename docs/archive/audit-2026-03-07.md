### 🟡 M-01: `FrozenDividendsRecovered` event — no test coverage

**File:** `test/WillChain.test.js`
**Severity:** Medium

The `FrozenDividendsRecovered` event (emitted when a recycled node had unclaimed dividends) is the **only new event without a dedicated test**. While the underlying functionality works (tested via other tests implicitly), no test verifies that:
1. The event emits with correct `abandonedNode` address
2. The event emits with correct `amount` value
3. The event does NOT emit when abandoned node had 0 unclaimed dividends

**Recommended test:**
```javascript
it("emits FrozenDividendsRecovered when recycled node had unclaimed dividends", async function () {
  // node1 registers, earns dividends, then goes inactive without claiming
  await phoenix.transfer(node1.address, ethers.parseEther("100000"));
  await phoenix.connect(node1).confirmActivity();
  
  // Create dividends by recycling node2
  await phoenix.transfer(node2.address, ethers.parseEther("1000"));
  await phoenix.connect(node2).confirmActivity();
  await time.increase(TOTAL_TIMEOUT + 1);
  await phoenix.connect(maintainer).recycleInactiveNode(node2.address);
  
  // node1 has unclaimed dividends now
  const pending = await phoenix.pendingDividends(node1.address);
  expect(pending).to.be.gt(0n);
  
  // node1 goes inactive and gets recycled
  await phoenix.connect(node1).confirmActivity(); // reset timer
  await time.increase(TOTAL_TIMEOUT + 1);
  
  await expect(phoenix.connect(maintainer).recycleInactiveNode(node1.address))
    .to.emit(phoenix, "FrozenDividendsRecovered")
    .withArgs(node1.address, anyValue);
});
```

### 🟡 M-02: `DividendsBurnedNoEligibleHolders` — test only verifies ABI, not emission

**File:** `test/WillChain.test.js:2393-2421`
**Severity:** Medium

The test for `DividendsBurnedNoEligibleHolders` only checks that the event exists in the ABI (`phoenix.interface.fragments`), but does **not** actually trigger the `eligibleSupply == 0` path and verify the event emission. The test comments explain why this is hard: the deployer (owner) is always registered, so `eligibleSupply > 0`.

**Scenario to reach `eligibleSupply == 0`:**
```javascript
it("actually triggers DividendsBurnedNoEligibleHolders emission", async function () {
  // 1. Owner transfers ALL tokens to node1 (unregistered)
  await phoenix.transfer(node1.address, INITIAL_SUPPLY);
  // 2. Owner has 0 balance. node1 is unregistered.
  // 3. node1 registers and goes abandoned
  await phoenix.connect(node1).confirmActivity();
  await time.increase(TOTAL_TIMEOUT + 1);
  // 4. At recycle time: eligibleSupply = totalSupply - node1_balance - contract_balance - unregistered
  //    = totalSupply - totalSupply - 0 - 0 = 0 (owner has 0, no other registered holders)
  //    BUT: owner is registered with 0 balance, so eligibleSupply still counts owner...
  //    Actually eligibleSupply = totalSupply - balanceOf(_from) - balanceOf(contract) - totalUnregisteredSupply
  //    = totalSupply - node1_balance - 0 - 0 = 0 ✓ (node1's balance IS _from, subtracted directly)
  await expect(phoenix.connect(maintainer).recycleInactiveNode(node1.address))
    .to.emit(phoenix, "DividendsBurnedNoEligibleHolders");
});
```

Owner has 0 balance, so they contribute 0 to eligibleSupply. `_from` (node1) is subtracted directly. `totalUnregisteredSupply` is 0 (node1 is registered → `everRegistered = true` at this point, but after `delete nodeStates`, `_isUnregistered` returns true... need to verify the exact flow). This may require a more careful setup.

### 🟡 M-03: `renounceOwnership()` is still callable

**File:** `WillChain.sol` (inherited from `Ownable2Step`)
**Severity:** Medium

`Ownable2Step` inherits `renounceOwnership()` from `Ownable`. If the owner accidentally calls this, **all admin functions are permanently disabled**:
- `proposeTreasuryChange` / `executeTreasuryChange` / `cancelTreasuryChange`
- `recoverDividendDust`

The protocol would still function (no pause mechanism), but treasury fees would accumulate with the old treasury address forever, and dust recovery becomes impossible.

**Fix (optional — depending on design intent):**
```solidity
/// @dev Disable renounceOwnership to prevent accidental permanent loss of admin access.
function renounceOwnership() public pure override {
    revert("Ownership renouncement disabled");
}
```

**Alternative:** Keep it if you want the option to fully decentralize the protocol by renouncing ownership post-launch. Document the decision either way.

### 🟢 L-01: `proposeTreasuryChange` can be overwritten (by design, but no event for overwrite)

**File:** `WillChain.sol:761-766`
**Severity:** Low

Calling `proposeTreasuryChange()` twice overwrites the first proposal without emitting a cancellation event. The second `TreasuryChangeProposed` event implicitly cancels the first, but off-chain monitoring tools have no explicit signal that the first proposal was superseded.

**Current behavior:**
```
propose(A) → TreasuryChangeProposed(A, t1)
propose(B) → TreasuryChangeProposed(B, t2)  ← A is silently gone
```

**Recommendation:** Either emit `TreasuryChangeCancelled(A)` before the new proposal, or document this as expected behavior. The test at line 2620 already covers this — it's a design decision, not a bug.
