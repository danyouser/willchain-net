## 🐛 КРИТИЧНІ БАГИ

### 1. `TransferModal.tsx` — виклик side-effects поза `useEffect`
**Файл:** `frontend-react/src/components/modals/TransferModal.tsx`, рядки 66–77

```tsx
// ❌ ПРОБЛЕМА: виконується під час рендеру, а не в useEffect
if (isSuccess) {
  showNotification(...)   // setState під час рендеру
  reset()
  setRecipient('')
  setAmount('')
  onClose()
}
```

**Чому критично:** Виклик `showNotification()`, `reset()`, `setRecipient()`, `setAmount()`, `onClose()` відбувається під час рендеру. React забороняє змінювати state під час рендеру — це призводить до попередження `Cannot update a component while rendering a different component` і некоректної поведінки.

На відміну від цього, `TimeCard.tsx` та `SuccessorCard.tsx` коректно використовують `useEffect([isSuccess, ...])`.

**Виправлення:**
```tsx
useEffect(() => {
  if (isSuccess) {
    showNotification({ type: 'success', ... })
    reset()
    setRecipient('')
    setAmount('')
    onClose()
  }
}, [isSuccess, ...])
```

### 2. Розбіжність ABI — поле `serviceTier` у `getNodeState`
**Файли:** `frontend-react/src/config/contract.ts` vs `contracts/PhoenixLegacy.sol`

Смарт-контракт повертає `string memory serviceTier`, але React ABI визначає це як `uint8`:

```ts
// contract.ts рядок 86 — НЕПРАВИЛЬНО:
{ name: 'serviceTier', type: 'uint8' }

// PhoenixLegacy.sol рядок 386 — ПРАВИЛЬНО:
string memory serviceTier
```

**Підтвердження з тестів** (`test/PhoenixLegacy.test.js`, рядок 591):
```js
expect(state.serviceTier).to.equal("Legacy Vault"); // рядок, не число
```

**Наслідок:** `useNodeState.ts` неправильно декодує `serviceTier` як число. Логіка `tierNames[serviceTier]` поверне `undefined`.

**Виправлення:** `{ name: 'serviceTier', type: 'string' }` + прибрати `tierNames[]` lookup.

### 3. Розбіжність ABI — структура `getNetworkStatistics`
**Файли:** `frontend-react/src/config/contract.ts`, рядки 94–103 vs `contracts/PhoenixLegacy.sol`

React ABI очікує **6 полів**: `totalNodes, totalSupply, circulatingSupply, recycledToNetwork, removedFromCirculation, successfulTransfers`

Контракт повертає лише **4 поля**: `totalSupply_, recycledToNetwork, removedFromCirculation, successfulTransfers`

**Підтвердження з тестів** (`test/PhoenixLegacy.test.js`, рядок 718–720):
```js
const stats = await phoenix.getNetworkStatistics();
expect(stats.totalSupply_).to.equal(INITIAL_SUPPLY); // тільки 4 поля
```

**Наслідок:** `totalNodes` і `circulatingSupply` завжди `undefined`. `HeroSection.tsx` завжди показує `Active Nodes: 0`.

**Виправлення:** Привести ABI у відповідність — 4 поля.

### 14. `formatTokenAmount` — втрата точності для великих bigint
**Файл:** `frontend-react/src/hooks/useBalance.ts`, рядок 23

```ts
const formatted = Number(amount) / 1e18  // втрата точності при amount > 2^53 wei
```

1 мільярд токенів з 18 десятковими = 1e27 wei >> 2^53. Результат буде неточним.

**Виправлення:** `(amount / 10n**18n).toLocaleString('en-US')`

### 21. `phoenix_observer.py` — використовує синхронні Web3 виклики в async функціях
**Файл:** `scripts/observer/phoenix_observer.py`, рядки 150–152

```python
async def get_current_nonce(self, address: str) -> int:
    return self.w3.eth.get_transaction_count(address)  # ❌ синхронний виклик в async
```

`web3.py` не є async-native. Синхронні RPC виклики в `asyncio` блокують event loop, що руйнує паралелізм.

**Виправлення:** Використовувати `asyncio.get_event_loop().run_in_executor()` або AsyncWeb3.

### 22. `personal_observer.py` — не перевіряє чи `account.address == WALLET_ADDRESS`
**Файл:** `scripts/observer/personal_observer.py`, рядок 91

```python
account = w3.eth.account.from_key(PRIVATE_KEY)
```

Скрипт використовує `PRIVATE_KEY` для підписання, але моніторить `WALLET_ADDRESS`. Якщо вони не співпадають — скрипт буде підтверджувати активність **іншого** гаманця. Немає жодної перевірки що `account.address == WALLET_ADDRESS`.
